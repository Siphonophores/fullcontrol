### Brim (new cell 5 in notebook)
A radial brim is prepended to the main print steps. Since the shape is
defined in polar coordinates, adding `EW * loop` to the radius is equivalent
to Illustrator's "offset path" — exact perpendicular offset with no
approximation needed.

- Loops go **outermost to innermost** for a clean continuous connection into
  the first layer of the main print (no travel gap, no ooze risk)
- Uses the same ripple + star formula as the first layer (bulge term omitted
  — `sin(0) = 0` at z=0 anyway)
- No skew on brim loops
- Controlled by `brim_loops` variable (default: 3)

```python
for loop in range(brim_loops, 0, -1):  # outermost to innermost
    offset = EW * loop
    for t in range(int(layer_segs)):
        t_val = t / layer_segs
        a_now = t_val * tau - tau/4
        r_now = inner_rad + offset + \
            rip_depth * (...) + (tip_length * (...))
        brim_steps.append(fc.polar_to_point(brim_centre, r_now, a_now))
steps = brim_steps + steps
```

This approach works for any parameter combination in this notebook. It does
not generalise to non-polar shapes (those require a Clipper-style geometric
offset).

### Overhang preview (new cell 7 in notebook)
Colours each printed point by overhang severity, analogous to Cura's red
overhang highlight.

**Key insight — why naïve point comparison fails:**
The ripple design uses `ripples_per_layer + 0.5` oscillations per revolution,
which deliberately offsets the ripple phase by 0.5 per layer (peaks align with
valleys on adjacent layers for an interlocked texture). Comparing same-index
points across layers therefore always measures peak↔valley — a ~1 mm
horizontal jump that falsely flags every layer as a severe overhang.

**Correct approach — layer-average radius:**
Compute the average radius of each layer and compare to the previous layer's
average. This cancels the ripple oscillations (intentional texture) and detects
only macro shape expansion (bulge, star, inner radius change):

```python
avg_r_curr = mean distance from centre for all points in layer N
avg_r_prev = mean distance from centre for all points in layer N-1
delta_r    = avg_r_curr - avg_r_prev
angle      = degrees(atan2(EH, delta_r))  # 90° = straight up, 0° = horizontal
```

Thresholds (configurable variables):
- `overhang_threshold = 45°` → red (dangerous overhang)
- `warning_threshold  = 60°` → orange (borderline)
- `≥ 60°`                   → green (safe)

Only outward growth (`delta_r > 0`) is flagged; shrinking or stable radius is
always 90° (safe). The brim and first print layer are skipped from comparison
since they have no meaningful "previous" layer.

Exports `overhang_red_layers` and `overhang_orange_layers` for use by the
gcode guard cell.

### Gcode guard (top of gcode cell)
Prevents accidentally printing a design with dangerous overhangs. Three states:
- Overhang cell not yet run → `RuntimeError` ("Run the overhang preview cell
  first")
- Red layers detected → `RuntimeError` (blocks gcode save entirely)
- Orange layers only → warning printed, gcode proceeds

```python
if 'overhang_red_layers' not in dir():
    raise RuntimeError('Run the overhang preview cell before generating gcode.')
if overhang_red_layers > 0:
    raise RuntimeError(f'Gcode blocked: {overhang_red_layers} layer(s) exceed ...')
if overhang_orange_layers > 0:
    print(f'Warning: {overhang_orange_layers} borderline layer(s). Proceeding...')
```

## Print Estimation

### Print time and filament cell (new cell after gcode cell)
Parses the most recently generated `.gcode` file in the notebook's folder and
reports estimated print time and filament usage. FullControl has no built-in
estimation; this fills the gap.

**Filament:**
Sums all positive `E` axis movement (excludes retracts), then converts:
```
volume (mm³) = total_E_mm × π × (1.75/2)²
mass   (g)   = volume × PLA_density / 1000     # PLA ≈ 1.24 g/cm³
```

**Print time — Marlin trapezoidal model:**
The same algorithm Cura uses. Each segment has an entry speed, a peak speed,
and an exit speed constrained by jerk and acceleration. Short segments never
reach full speed; the model accounts for this correctly (unlike a naïve
distance/feedrate sum, which typically underestimates by ~15–20%).

```
Ender 3 params used:
  acceleration        = 500 mm/s²   (print moves)
  travel acceleration = 1000 mm/s²  (G0 moves)
  jerk                = 10 mm/s     (max instantaneous speed change at junctions)
```

For each segment:
- If `d_accel + d_decel ≤ distance` → trapezoid (accel + cruise + decel)
- Otherwise → triangle (no cruise phase; peak speed lower than feedrate)

Sample results on actual gcode:
| File            | Time    | Filament          |
|-----------------|---------|-------------------|
| ripple-texture-01 | 55 min | 1.3 g / 434 mm   |
| ripple-texture-03 | 1h 57m | 2.8 g / 942 mm   |

The cell automatically finds the newest `.gcode` file in the same directory,
so no path wiring is needed.
